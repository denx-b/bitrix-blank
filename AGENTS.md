# Инструкции для агентов

Этот репозиторий - шаблон проекта 1С-Битрикс с собственным Legacy-фреймворком поверх `local/php_interface`.
Главная договоренность: не превращать `init.php` в свалку процедурного кода.

## Базовая архитектура

- `local/php_interface/init.php` должен содержать только подключение фреймворка:
  ```php
  require_once __DIR__ . '/legacy/bootstrap.php';
  ```
- Не добавляйте бизнес-логику, регистрацию событий, функции, константы или `require`-цепочки прямо в `init.php`.
- Точка входа фреймворка - `local/php_interface/legacy/bootstrap.php`.
- `bootstrap.php` подключает:
  - `autoload.php` для PSR-4 автозагрузки;
  - `settings.php` для констант и настроек;
  - `vendor/autoload.php`, если он существует;
  - все PHP-файлы из `legacy/functions`;
  - все PHP-файлы из `legacy/eventHandlers`.

## Автозагрузка и классы

- Все классы проекта размещаются в `local/php_interface/legacy/classes`.
- Используйте namespace `Legacy\...`.
- Автозагрузка работает по правилу PSR-4:
  - `Legacy\Some\ClassName` -> `local/php_interface/legacy/classes/Some/ClassName.php`.
- Код пишется в стиле PSR-12.
- Новую логику размещайте в доменных директориях внутри `classes`, например:
  - `Agents` - агенты;
  - `Api` - API-инфраструктура и методы;
  - `Const` - классы с константами;
  - `Events` - классы-обработчики событий;
  - `Model` - модели и ORM-таблицы;
  - `Service` - сервисы и утилитарная бизнес-логика;
  - `Rest` - REST-интеграции.
- Если нужна новая категория, создавайте осмысленную директорию по ООП-назначению, а не складывайте код в общий файл.

## События Битрикса

- Регистрацию обработчиков событий размещайте в `local/php_interface/legacy/eventHandlers`.
- Реализацию обработчиков размещайте в классах, обычно в `Legacy\Events`.
- Принято называть файл регистрации по модулю Битрикса:
  - `main.php` для модуля `main`;
  - `iblock.php` для модуля `iblock`;
  - `sale.php` для модуля `sale`;
  - `crm.php` для модуля `crm`.
- Имя файла в `eventHandlers` - рекомендация для навигации, а не жесткое правило. Файл может называться иначе, и файлов может быть несколько.
- В файлах `eventHandlers` держите только регистрацию и минимальные комментарии. Бизнес-логику выносите в классы.
- Каждая регистрация обработчика события обязательно должна быть прокомментирована многострочным комментарием на 1-2 строки.
- Используйте обычный многострочный комментарий с одной звездочкой, не PHPDoc:
  ```php
  /*
   * Метод запрещает изменять элементы не авторам.
   */
  ```
- Метод обработчика должен соблюдать принцип единственной ответственности.
- Не объединяйте в один обработчик несколько независимых задач только ради уменьшения количества регистраций.
- В 1С-Битрикс регистрация обработчика не создает значимой нагрузки: она сохраняет описание обработчика в массив, а выполнение идет последовательно только при срабатывании события.
- Если на одно событие нужно повесить несколько разных бизнес-действий, регистрируйте несколько отдельных обработчиков с понятными комментариями и отдельными методами.
- Пример:
  ```php
  /*
   * Метод запрещает изменять элементы не авторам.
   */
  AddEventHandler('iblock', 'OnBeforeIBlockElementUpdate', [
      '\Legacy\Events\IblockHandlers',
      'onCheckPermissionBeforeUpdate',
  ]);
  ```

## Глобальные функции

- Глобальных функций лучше избегать.
- Если глобальная функция действительно нужна, размещайте ее в `local/php_interface/legacy/functions`.
- Рекомендуется называть файл так же, как функцию, например `custom_mail.php`.
- Все PHP-файлы из `legacy/functions` подключаются автоматически.
- Не добавляйте глобальные функции в `init.php`.

## Настройки и зависимости

- Общие константы проекта размещайте в `local/php_interface/legacy/settings.php`.
- Composer-зависимости подключаются автоматически из `local/php_interface/legacy/vendor/autoload.php`, если файл существует.
- Не добавляйте ручные подключения Composer-зависимостей в `init.php`.
- Проект рассчитан на PHP `>=8.1`; учитывайте это при выборе синтаксиса и зависимостей.

## API

- Запросы `/api/*` направляются в `local/php_interface/legacy/api.php` через `urlrewrite`.
- `api.php` нормализует маршрут и передает его в `Legacy\Api\ApiFactory::create(...)`.
- `ApiFactory` выбирает класс обработчика по карте `Legacy\Api\ApiRoutes::map()`.
- Если маршрут не найден, используется `Legacy\Api\Methods\Unknown`.
- Новые API-методы размещайте в `local/php_interface/legacy/classes/Api/Methods`.
- Новый маршрут добавляйте централизованно в `Legacy\Api\ApiRoutes::map()`.
- Пример соответствия:
  - `/api/news` -> `Legacy\Api\Methods\News`;
  - `/api/deploy/gitStatus` -> `Legacy\Api\Methods\Deploy\GitStatus`.

## Шаблоны

- В шаблонах не добавляйте `addCss` и `addJs` прямо внутри HTML-тега `<head>` как будто место вызова является местом вывода.
- Подключайте JS, CSS, favicon, viewport и другие head-ресурсы через `Bitrix\Main\Page\Asset` перед HTML-разметкой.
- `$APPLICATION->ShowHead()` должен оставаться внутри `<head>` и выводить уже зарегистрированные ресурсы.
- PHP-подготовку в `header.php` размещайте до HTML-содержимого шаблона.

## Практические правила для правок

- Перед изменениями проверьте существующие примеры в соседних файлах и следуйте текущему стилю проекта.
- Не смешивайте регистрацию события и реализацию обработчика в одном процедурном файле.
- Не создавайте новые глобальные функции, если можно решить задачу классом или сервисом.
- Не добавляйте код в `init.php`, кроме уже существующего подключения `legacy/bootstrap.php`.
- Для новых классов соблюдайте соответствие namespace и пути файла.
- Для API-методов не обходите `ApiRoutes` и `ApiFactory`.
- Для изменений в шаблонах учитывайте договоренности из `README.md` про `Asset` и `$APPLICATION->ShowHead()`.
